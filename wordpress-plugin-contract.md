# LinkerFlow WordPress Plugin API Contract (linkerflow/v1)

This document is the frozen interface contract between the LinkerFlow WordPress plugin and the LinkerFlow application. It is the single source of truth that both sides build against, and it doubles as public API documentation for site owners and agencies who want to understand or audit what the plugin exposes.

- **Provider:** WordPress (self-hosted, WordPress.org)
- **Namespace:** `linkerflow/v1`
- **Base URL:** `https://<your-site>/wp-json/linkerflow/v1`
- **Transport:** HTTPS only
- **Content format:** WordPress native HTML (`post_content`), including Gutenberg block markup
- **Plugin targets:** WordPress 6.5+, PHP 7.4+, no external dependencies (WordPress core APIs only)

## Scope and guarantees

The plugin only ever:

- reads published posts, pages, and selected custom post types,
- reads the resolved permalink and language of those items,
- writes internal link anchors back into `post_content`.

The plugin never reads or writes users, settings, other plugins, or any field other than `post_content`. Every write creates a native WordPress revision, so changes are reversible from the WordPress editor.

## Authentication

After connection (see Handshake), LinkerFlow holds an opaque shared secret. Every request from LinkerFlow to the plugin carries it:

```
Authorization: Bearer <secret>
```

The plugin validates the secret in `permission_callback` on every endpoint. A missing or wrong secret returns `401`. The secret is stored encrypted at rest on the LinkerFlow side (AES-256-GCM) and as an opaque option on the WordPress side.

## Handshake (connection)

Connection is plugin-initiated and proves site ownership through a nonce round-trip. No manual key copy-paste.

1. In WP Admin, the site owner clicks "Connect to LinkerFlow".
2. The plugin generates a one-time `nonce`, then redirects the browser to:
   `https://app.linkerflow.io/api/wordpress/connect?site_url=<site>&nonce=<nonce>&state=<state>`
3. The user approves the connection in LinkerFlow (authenticated session). The `state` parameter guards against CSRF.
4. LinkerFlow generates the shared `secret` and calls the plugin to confirm, server to server over HTTPS: `POST /connect` with `{ nonce, secret }`.
5. The plugin validates that the `nonce` matches the one it generated (only this site holds it), stores the `secret`, and returns success.
6. LinkerFlow stores the encrypted secret and the site REST base URL, and creates the site with `provider = wordpress`.

The secret is generated by LinkerFlow and delivered to the plugin only in this server-to-server call, never through the browser or a query string, so it cannot leak in logs or referrers. The nonce proves the plugin instance is the genuine one for that site.

If LinkerFlow cannot reach the plugin endpoint at `site_url` (localhost, firewall, HTTP auth), the handshake fails with a clear "site unreachable" message and leaves no partial connection behind.

## Conventions

- **Field names:** WordPress-native `snake_case`, returned verbatim. LinkerFlow maps to its own shapes at its adapter boundary.
- **Pagination:** query params `page` (1-based) and `per_page`. Responses include `X-WP-Total` and `X-WP-TotalPages` headers.
- **Dates:** `post_modified` is a WordPress datetime string.
- **Permalinks:** always the value of `get_permalink()`, never reconstructed by LinkerFlow.
- **Errors:** standard WordPress REST error body.

### Error format

```json
{
  "code": "linkerflow_unauthorized",
  "message": "Invalid or missing credential.",
  "data": { "status": 401 }
}
```

LinkerFlow treats `401`, `403`, and a `404`-gone as permanent (disconnect), and `5xx` or timeouts as transient (retry).

## Endpoints

### POST /connect

Confirms the handshake. Validates the `nonce` against the one the plugin generated, then stores the LinkerFlow-supplied secret.

Request body:

```json
{ "nonce": "<nonce>", "secret": "<linkerflow-generated-secret>" }
```

Response `200`:

```json
{ "connected": true, "site_url": "https://example.com", "account_name": "Example Blog" }
```

Errors: `401` (nonce invalid or expired).

This is the only unauthenticated endpoint (it bootstraps the credential); it is gated by the one-time nonce instead of the Bearer secret.

### GET /post-types

Lists the public content types available to manage. Used by onboarding so the user can choose what to connect.

Response `200`:

```json
{
  "post_types": [
    { "slug": "post", "label": "Posts" },
    { "slug": "page", "label": "Pages" },
    { "slug": "portfolio", "label": "Portfolio" }
  ]
}
```

### GET /languages

Reports multilingual configuration. The language is read from WPML or Polylang APIs, never inferred. Returns an empty list (or a single default) for monolingual sites.

Response `200`:

```json
{
  "multilingual": true,
  "plugin": "polylang",
  "default": "en",
  "languages": [
    { "code": "en", "label": "English", "subdirectory": "/", "default": true },
    { "code": "fr", "label": "Francais", "subdirectory": "/fr/", "default": false }
  ]
}
```

`subdirectory` reflects the real URL form (subdirectory or subdomain) as configured.

### GET /posts

Returns published items for ingestion. Paginated. Supports incremental refresh via `modified_after`.

Query params:

| Param | Required | Description |
|---|---|---|
| `post_type` | yes | A slug from `GET /post-types`. |
| `page` | no | 1-based page number (default 1). |
| `per_page` | no | Items per page. |
| `status` | no | Always `publish`; other statuses are excluded. |
| `modified_after` | no | ISO datetime; returns only items with `post_modified` newer than this (incremental crawl). |

Response `200` (array; pagination in headers):

```json
[
  {
    "id": 42,
    "permalink": "https://example.com/guide-seo-2026/",
    "title": "Guide SEO 2026",
    "post_type": "post",
    "status": "publish",
    "locale": "fr",
    "post_content": "<!-- wp:paragraph --><p>...</p><!-- /wp:paragraph -->",
    "post_modified": "2026-06-01T10:30:00"
  }
]
```

Page-builder behavior: items managed by an unsupported page builder (detected via markers such as `_elementor_data` or `et_pb_`) are excluded from results and reported with a reason, so the rest of the site still ingests.

Meta description: not yet returned. A planned follow-up will have the plugin expose the available meta-description sources per post (Yoast `_yoast_wpseo_metadesc`, Rank Math `rank_math_description`, or the excerpt) so onboarding can let the user pick the source per post type, mirroring the Webflow/Wix meta-description field selection. Until then LinkerFlow stores a null meta description for WordPress pages.

### GET /count

Returns the aggregated count of published items, computed in a single call (`wp_count_posts`), so LinkerFlow does not page through everything just to size the site.

Response `200`:

```json
{ "published": 318 }
```

### POST /posts/{id}

Writes updated `post_content` for one item. Used to publish, edit, remove, and batch-fix internal links. The update goes live immediately and creates a WordPress revision.

Request body:

```json
{ "post_content": "<!-- wp:paragraph --><p>... <a href=\"...\">anchor</a> ...</p><!-- /wp:paragraph -->" }
```

Response `200`:

```json
{ "id": 42, "updated": true, "revision_id": 1057 }
```

Errors: `404` (item not found), `409` or similar if the item is read-only.

Only link anchors are changed by LinkerFlow; the rest of the content and the block markup are preserved.

## Versioning

The namespace carries the version (`linkerflow/v1`). Breaking changes ship under a new namespace (`linkerflow/v2`) so older plugin and application versions keep working. Additive, non-breaking fields may be added to `v1` responses.

## Security summary

- HTTPS only; Bearer secret on every request, validated in `permission_callback`.
- Handshake uses a one-time nonce plus a `state` parameter and verifies `site_url` ownership.
- Secret encrypted at rest on the LinkerFlow side.
- Least privilege: published content read plus `post_content` write only.
- `uninstall.php` removes the plugin's stored options on deletion.
