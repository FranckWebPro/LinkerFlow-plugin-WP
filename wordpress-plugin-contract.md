# LinkerFlow WordPress Plugin API Contract (linkerflow/v1)

This document is the frozen interface contract between the LinkerFlow WordPress plugin and the LinkerFlow application. It is the single source of truth that both sides build against, and it doubles as public API documentation for site owners and agencies who want to understand or audit what the plugin exposes to the LinkerFlow service.

- **Provider:** WordPress (self-hosted, WordPress.org)
- **Namespace:** `linkerflow/v1`
- **Base URL:** `https://<your-site>/wp-json/linkerflow/v1`
- **Transport:** HTTPS only
- **Content format:** WordPress native HTML (`post_content`), including Gutenberg block markup
- **Plugin targets:** WordPress 6.5+, PHP 7.4+, no external dependencies (WordPress core APIs only)

## Build and release

This is the plugin source repo (`LinkerFlow-WP/`). The two shipped zips are build artifacts and must never be edited directly. To release a change: edit the source under `linkerflow/`, bump the version in `linkerflow.php` (the `Version:` header and `LINKERFLOW_VERSION`) and `readme.txt` (`Stable tag` plus a changelog entry), run `./build.sh`, then copy `linkerflow.zip` to the app repo's `public/linkerflow-wp.zip` and `linkerflow-test.zip` to `public/linkerflow-wp-test.zip`. `build.sh` swaps only the `APP_CONNECT_URL` constant per build (prod `app.linkerflow.io`, test `staging.linkerflow.io`) and restores the source to the prod default afterward; that constant is the sole difference between the two zips. This contract file exists in both the plugin repo and the app repo and must be kept in sync.

## Scope and guarantees

The plugin only ever:

- reads published, non-password-protected posts, pages, and selected custom post types,
- reads the resolved permalink and language of those items,
- writes internal link anchors back into `post_content`.

The plugin never reads or writes users, settings, other plugins, private posts, drafts, password-protected posts, or any field other than `post_content`. Writes use WordPress' native post update flow and create a revision when revisions are enabled for the post type.

## Authentication

After connection (see Handshake), the LinkerFlow application holds an opaque shared secret. Every request from the LinkerFlow service to the plugin carries it:

```
Authorization: Bearer <secret>
```

The plugin validates the secret in `permission_callback` on every endpoint. A missing or wrong secret returns `401`. The secret is stored encrypted at rest by the LinkerFlow application (AES-256-GCM) and as an opaque option on the WordPress side.

## Handshake (connection)

Connection is plugin-initiated and proves site ownership through a nonce round-trip. No manual key copy-paste.

1. In WP Admin, the site owner clicks "Connect to LinkerFlow".
2. The plugin generates a one-time `nonce`, then redirects the browser to:
   `https://app.linkerflow.io/api/wordpress/connect?site_url=<site>&nonce=<nonce>&state=<state>`
3. The user approves the connection in the LinkerFlow application (authenticated session). The `state` parameter guards against CSRF.
4. The LinkerFlow application generates the shared `secret` and calls the plugin to confirm, server to server over HTTPS: `POST /connect` with `{ nonce, secret }`.
5. The plugin validates that the `nonce` matches the one it generated (only this site holds it), stores the `secret`, and returns success.
6. The LinkerFlow application stores the encrypted secret and the site REST base URL, and creates the site with `provider = wordpress`.

The secret is generated by the LinkerFlow application and delivered to the plugin only in this server-to-server call, never through the browser or a query string, so it cannot leak in logs or referrers. The nonce proves the plugin instance is the genuine one for that site.

If the LinkerFlow application cannot reach the plugin endpoint at `site_url` (localhost, firewall, HTTP auth), the handshake fails with a clear "site unreachable" message and leaves no partial connection behind.

## Conventions

- **Field names:** WordPress-native `snake_case`, returned verbatim. The LinkerFlow application maps to its own shapes at its adapter boundary.
- **Pagination:** query params `page` (1-based) and `per_page`. Responses include `X-WP-Total` and `X-WP-TotalPages` headers.
- **Dates:** `post_modified` is a WordPress datetime string.
- **Permalinks:** always the value of `get_permalink()`, never reconstructed by the LinkerFlow application.
- **Errors:** standard WordPress REST error body.

### Error format

```json
{
  "code": "linkerflow_unauthorized",
  "message": "Invalid or missing credential.",
  "data": { "status": 401 }
}
```

The LinkerFlow application treats `401`, `403`, and a `404`-gone as permanent (disconnect), and `5xx` or timeouts as transient (retry).

## Endpoints

### POST /connect

Confirms the handshake. Validates the `nonce` against the one the plugin generated, then stores the LinkerFlow-application-supplied secret.

Request body:

```json
{ "nonce": "<nonce>", "secret": "<linkerflow-generated-secret>" }
```

Response `200`:

```json
{ "connected": true, "site_url": "https://example.com", "account_name": "Example Blog" }
```

Errors: `401` (nonce invalid or expired).

This is the only unauthenticated endpoint (it bootstraps the credential); it is gated by the one-time nonce instead of the Bearer secret.

### GET /post-types

Lists the public content types available to manage. Used by LinkerFlow onboarding so the user can choose what to connect.

Page-builder and WordPress-internal post types that are `public` but not editorial content are excluded from the list (and from crawl validation), so they never appear as selectable collections: `attachment`, Elementor (`elementor_library`, `e-floating-buttons`, `e-landing-page`), Divi layout templates (`et_pb_layout`, `et_template`, `et_header_layout`, `et_footer_layout`, `et_body_layout`), and block/template types (`wp_block`, `wp_template`, `wp_template_part`, `wp_navigation`). The deny list is filterable via the `linkerflow_excluded_post_types` WordPress filter.

The response also reports `meta_sources`: the meta-description sources detected on the site. SEO plugins store the description per post site-wide, so onboarding surfaces a single meta-source selector rather than a per-collection field. `yoast` is present when Yoast SEO is active (`WPSEO_VERSION`), `rankmath` when Rank Math is active (`RANK_MATH_VERSION`), and `excerpt` is always offered last. SEO plugins are listed first so the first entry is the sensible default.

Response `200`:

```json
{
  "post_types": [
    { "slug": "post", "label": "Posts" },
    { "slug": "page", "label": "Pages" },
    { "slug": "portfolio", "label": "Portfolio" }
  ],
  "meta_sources": [
    { "slug": "yoast", "label": "Yoast SEO" },
    { "slug": "excerpt", "label": "Excerpt" }
  ]
}
```

### GET /languages

Reports multilingual configuration. The language is read from WPML or Polylang APIs, never inferred. Returns an empty list (or a single default) for monolingual sites.

Response `200`:

```json
{
  "multilingual": true,
  "plugin": "polylang",
  "default": "en",
  "languages": [
    { "code": "en", "label": "English", "subdirectory": "/", "default": true },
    { "code": "fr", "label": "Francais", "subdirectory": "/fr/", "default": false }
  ]
}
```

`subdirectory` reflects the real URL form (subdirectory or subdomain) as configured.

### GET /posts

Returns published items for ingestion. Paginated. Supports incremental refresh via `modified_after`.

Query params:

| Param | Required | Description |
|---|---|---|
| `post_type` | yes | A slug from `GET /post-types`. |
| `page` | no | 1-based page number (default 1). |
| `per_page` | no | Items per page. |
| `status` | no | Always `publish`; other statuses and password-protected posts are excluded. |
| `modified_after` | no | ISO datetime; returns only items modified after this instant (incremental crawl). Compared against `post_modified_gmt` in UTC, so site timezone never shifts the window. |
| `meta_source` | no | A slug from `meta_sources` (`yoast`, `rankmath`, `excerpt`). Selects which field each item's `meta_description` is read from. Omitted = `meta_description` is null. |

Multilingual behavior: when WPML or Polylang is active, results span every language (each item carries its `locale`), not just the site default language, so the application can segment posts per language. The site default language is used when neither plugin is active. Only subdirectory language URLs are supported; subdomain and separate-domain configurations are treated as separate sites.

Response `200` (array; pagination in headers):

```json
[
  {
    "id": 42,
    "permalink": "https://example.com/guide-seo-2026/",
    "title": "Guide SEO 2026",
    "post_type": "post",
    "status": "publish",
    "locale": "fr",
    "post_content": "<!-- wp:paragraph --><p>...</p><!-- /wp:paragraph -->",
    "post_modified": "2026-06-01T10:30:00",
    "meta_description": "Guide complet du linking interne."
  }
]
```

Page-builder behavior: Elementor and Divi 4 (shortcode format) pages are supported. The plugin flattens their content to HTML on read (Elementor `text-editor` widgets and Divi `[et_pb_text]` modules, plus headings and builder button/call-to-action links so the link graph sees builder-built navigation) and writes approved links back into the originating text widget or text module via an anchor diff, never overwriting the rest of the layout. Elementor writes update the `_elementor_data` post meta and flush the Elementor CSS cache; Divi writes update `post_content` and clear the Divi static CSS cache. Pages built with a still-unsupported builder (WPBakery `[vc_row`, Avada `[fusion_builder`, Divi 5 block format `<!-- wp:divi/` or Divi builder enabled without Divi 4 shortcodes) are excluded from results and reported with a reason, so the rest of the site still ingests.

Meta description: each item carries `meta_description`, resolved from the `meta_source` the user picked at onboarding. `yoast` reads `_yoast_wpseo_metadesc`, `rankmath` reads `rank_math_description`, `excerpt` reads `post_excerpt`. SEO-plugin template variables (e.g. `%%title%%`) are expanded when the plugin's helper is available. When the chosen source is empty for a post it falls back to the excerpt, then to null. The available sources are advertised site-wide via `meta_sources` on `GET /post-types`; the crawl passes the selected slug as `meta_source` on `GET /posts`.

### GET /count

Returns the count of published, non-password-protected items so the LinkerFlow application does not page through everything just to size the site.

Query params:

| Param | Required | Description |
|---|---|---|
| `post_type` | no | A slug from `GET /post-types`. Omitted = all supported public post types. |
| `lang` | no | A language slug from `GET /languages`. When set (WPML/Polylang active), counts only that language. Omitted means **no language filter**: in a REST context WPML/Polylang count posts across all languages, which inflates a multilingual primary count, so the application passes the default language slug explicitly for the primary locale and never relies on the omitted default. Omit it only for monolingual sites (one language = the whole site). Page limits are per language, so each locale is sized independently. |

Response `200`:

```json
{ "published": 318 }
```

### GET /posts/{id}

Returns the current `post_content` for one published, non-password-protected item so the LinkerFlow application applies a link on top of the live content rather than a stale crawl snapshot. Same guards as the write endpoint.

Response `200`:

```json
{ "id": 42, "permalink": "https://example.com/guide/", "post_content": "<!-- wp:paragraph --><p>...</p><!-- /wp:paragraph -->", "status": "publish", "locale": "fr" }
```

Errors: `404` (item not found / unpublished / password-protected / unsupported type), `409` (read-only, e.g. content from a still-unsupported page builder).

### POST /posts/{id}

Writes updated `post_content` for one published, non-password-protected item. Used to publish, edit, remove, and batch-fix internal links. The update goes live immediately and creates a WordPress revision when revisions are enabled for the post type.

Request body:

```json
{ "post_content": "<!-- wp:paragraph --><p>... <a href=\"...\">anchor</a> ...</p><!-- /wp:paragraph -->" }
```

Response `200`:

```json
{ "id": 42, "updated": true, "revision_id": 1057 }
```

Errors: `404` (item not found), `409` or similar if the item is read-only, `422` when a page-builder write finds no matching text block to change (nothing was written; the application marks the change failed instead of assuming it went live).

Only link anchors are changed by the LinkerFlow service; the rest of the content and the block markup are preserved.

## Versioning

The namespace carries the version (`linkerflow/v1`). Breaking changes ship under a new namespace (`linkerflow/v2`) so older plugin and application versions keep working. Additive, non-breaking fields may be added to `v1` responses.

## Security summary

- HTTPS only; Bearer secret on every request, validated in `permission_callback`.
- Handshake uses a one-time nonce plus a `state` parameter to verify that the callback matches a connection initiated from this WordPress site.
- Secret encrypted at rest by the LinkerFlow application.
- Least privilege: published, non-password-protected content read plus `post_content` write only.
- `uninstall.php` removes the plugin's stored options on deletion.
